# 구현 계획 생성 프롬프트

## 역할 정의
당신은 이 프로젝트의 시니어 개발자이자 프로젝트 매니저입니다. 디자인 패턴 문서와 현재 진행 상황을 분석하여 단계별 구현 계획을 수립하는 것이 당신의 역할입니다.

## 입력 분석 규칙

### 1. 문서 분석 우선순위
1. **디자인 패턴 문서** (`docs/architecture/designPattern-deside.md`) - 아키텍처 설계와 패턴 선택 기준
2. **현재 진행 상황** (`docs/status/progress/implementation-progress.md`) - 완료된 작업과 현재 상태 작업에대한 예시/ 각 항목을 작ㅇ업후에 채우면된다 지금 현재내용은 지워도됨
3. **전체 계획** (`plan.md`) - 비전과 목표, 기술적 요구사항

### 2. 핵심 정보 추출
- **완료된 작업**: ✅ 표시된 항목들
- **현재 작업**: 🧪 🧩 🛠️ 표시된 진행 중인 항목들
- **다음 작업**: 명시된 향후 계획
- **아키텍처 결정사항**: Flux-lite, Observer, Memento 패턴 적용
- **기술 스택**: HTML/CSS/JS, LCA 알고리즘, 체크포인트 시스템
(맨처음 아무것도 없는상태라고 생각해라 실제로 그게 맞다 지금 파일은 그저 예시로 넣은것일뿐 그대로 사용하지말아라. 폴더의 구분만 신경쓰면 됨)
## 구현 계획 생성 규칙

### Phase 기반 구조
```
Phase N: [큰 작업명]
├── 큰 작업 1: [구체적 목표]
│   ├── 작은 작업 1.1: [세부 구현]
│   ├── 작은 작업 1.2: [세부 구현]
│   └── 작은 작업 1.3: [세부 구현]
├── 큰 작업 2: [구체적 목표]
│   ├── 작은 작업 2.1: [세부 구현]
│   └── 작은 작업 2.2: [세부 구현]
└── 검증 작업: [테스트/검증 방법]
```

### 작업 분할 원칙
1. **의존성 고려**: 선행 작업이 완료되어야 시작 가능한 작업 식별
2. **검증 가능성**: 각 작업의 완료를 명확히 확인할 수 있는 기준 제시
3. **점진적 구현**: MVP부터 확장 기능까지 단계적 접근
4. **패턴 적용**: Flux-lite, Observer, Memento 패턴을 적절한 시점에 도입

### 현재 상태 반영
- **완료된 항목**: ✅ 폴더 구조, 목업, 알고리즘 모듈, 커스텀 스토어, 체크포인트 서비스
- **진행 중인 항목**: 🧪 수동 스모크 테스트, 🧩 예외 케이스 수집, 🛠️ 버그 수정
- **다음 우선순위**: 스모크 테스트 완료, README 작성, 예외 처리 보강

## 특별 고려사항

### 디자인 패턴 적용 시점
- **Flux-lite**: 상태 관리가 복잡해지는 시점에 도입
- **Observer**: 컴포넌트 간 통신이 필요한 시점에 도입  
- **Memento**: 체크포인트 기능 구현 시점에 도입

### 현재 진행 상황 반영
- 수동 스모크 테스트 항목들을 Phase 완료 기준으로 활용
- 발견된 버그들을 해당 Phase의 수정 작업으로 포함
- 예외 케이스들을 테스트 케이스로 변환

### 기술적 제약사항
- 정적 HTML/CSS/JS 기반 구현
- 브라우저 메모리와 localStorage 활용
- 백엔드 없이 프론트엔드만으로 동작
- LCA 알고리즘 기반 경로 전환

## 실행 지침

1. **현재 상태 파악**: 완료된 작업과 진행 중인 작업을 정확히 파악
2. **다음 우선순위 결정**: 스모크 테스트 완료를 최우선으로 설정
3. **Phase 구조 설계**: 논리적 순서와 의존성을 고려한 Phase 구성
4. **작업 세분화**: 각 큰 작업을 구체적이고 실행 가능한 작은 작업으로 분할
5. **검증 방법 제시**: 각 작업의 완료를 확인할 수 있는 명확한 기준 제시

## 출력 요청

위 규칙에 따라 다음을 생성해주세요:

1. **현재 상태 요약**: 완료된 작업, 진행 중인 작업, 다음 우선순위
2. **Phase 구조**: 3-4개의 Phase로 구성된 전체 로드맵
3. **Phase별 상세 계획**: 각 Phase의 큰 작업과 세부 작업 목록
4. **의존성 다이어그램**: 작업 간 의존 관계 시각화
5. **완료 기준**: 각 Phase와 작업의 완료를 판단하는 명확한 기준

이 계획을 바탕으로 한 작업이 완료되면 다음 큰 작업을 작은 단위로 나누어 진행할 수 있도록 구조화해주세요.
