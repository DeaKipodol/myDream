## 이슈 심각도 정의

- **P0 (Critical)**: 프로젝트 전체 중단 또는 크리티컬 패스 블로킹
- **P1 (High)**: Phase 지연 또는 주요 기능 영향
- **P2 (Medium)**: 개선 필요하지만 일정에 미치는 영향 제한적
- **P3 (Low)**: 사소한 개선 사항, 향후 처리 가능

---

## 이슈 상태 정의

- **발견 (Discovered)**: 이슈가 처음 발견되고 기록됨
- **분석중 (Analyzing)**: 원인 분석 및 영향 범위 조사 중
- **대응중 (Responding)**: 대응 계획 실행 중 (Phase 재편성, 담당자 재배정 등)
- **검증중 (Verifying)**: 대응 완료 후 효과 검증 중
- **해결 (Resolved)**: 이슈 완전 해결 및 아카이브

---

## 이슈 보고 프로세스

1. **이슈 발견**: 담당자 또는 PM이 이슈 감지
2. **이슈 기록**: 본 문서에 새 행 추가 (ID, 제목, 심각도, 발견일 등)
3. **원인 분석**: 탐정 모드 활성화, 근본 원인 파악
4. **대응 계획 수립**: Phase 재편성, 담당자 재배정, 기술 총괄자 요청 등
5. **대응 실행**: 계획에 따라 조치
6. **검증**: 이슈 해결 여부 확인
7. **아카이브**: 해결된 이슈를 "해결된 이슈" 섹션으로 이동

## 자동 모니터링 항목

PM은 다음 징후를 일일 모니터링하여 조기 이슈를 탐지합니다:

- **지연 징후**: 예상 완료일 대비 실제 진행률 80% 미만
- **담당자 보고 누락**: 2일 이상 진행 보고 없음
- **문서 미작성**: Phase 완료 기준에 명시된 문서가 작성되지 않음
- **기술 충돌 가능성**: 프론트-백엔드 API 명세 불일치 감지
- **범위 확장**: 원래 개발지시서에 없던 작업 추가 감지
- **의존성 블로커**: 선행 Phase 미완료로 다음 Phase 시작 불가
